# UX Planning Artifact - clmm-v2

**Author:** BMad  
**Date:** 2026-03-21

## 1. Primary Personas And Usage Context

### Primary Persona: Active Orca LP On Mobile

- experienced Solana CLMM participant
- checks positions throughout the day from a phone
- does not need CLMM education, but does need fast operational clarity
- values speed, directional clarity, and confidence before signing

### Secondary Persona: Mobile-Heavy DeFi Operator

- manages multiple activities during the day and reacts to alerts in short sessions
- often enters from a push notification rather than from app browsing
- needs an interface that can be understood in seconds, not minutes

### Support / Review Persona

- reviews past triggers, execution attempts, and failure states
- mostly uses desktop PWA for history, auditability, and troubleshooting
- does not need a broad control surface, only accurate operational records

### Usage Context

- most important sessions are urgent, short, and interrupt-driven
- the dominant context is mobile alert -> app open -> preview -> sign
- desktop is a secondary context for review, history, and supported manual execution
- users are operating under market movement, stale quotes, and transaction uncertainty

## 2. Information Architecture

### Primary Objects

- Wallet
- Supported Positions
- Position Status
- Exit Policy
- Actionable Alerts
- Execution Preview
- Execution State
- Off-Chain History

### IA Principles

- lead with the affected position and current range state, not a dashboard summary
- let alerts and position details converge into the same execution preview surface
- keep history and result review separate from action-taking, but reachable in one tap
- treat exit policy as a narrow guardrail definition, not a strategy builder

### Top-Level IA

- Positions
- Alerts
- History
- Wallet / App Settings

### Position Detail IA

- current range state
- breach direction or in-range state
- fixed MVP exit policy summary
- monitoring readiness and notification state
- most recent alert or execution state
- link to history for that position

## 3. Key Screens And Flows

### Flow 1: Connect Wallet

- launch app
- short value framing: LP Exit Guardrail, not automation
- connect supported wallet
- confirm wallet connected
- request notification permission with plain-language explanation
- show monitoring readiness state

### Flow 2: View Supported CLMM Position

- land on Positions list
- show supported Orca positions only
- each row shows pool, range status, and whether action is required
- tap into Position Detail

### Flow 3: Understand Current Range Status

- Position Detail shows one primary state block:
  - In Range
  - Below Range
  - Above Range
  - Actionable Downside Trigger
  - Actionable Upside Trigger
- state block explains what that means in plain language
- for actionable states, the screen must pair the state label with the expected post-exit posture: downside -> SOL to USDC, upside -> USDC to SOL
- if out of range, show current likely one-sided posture

### Flow 4: Configure Or Review Exit Policy

- MVP UX centers on review, not custom strategy authoring
- users can review the fixed directional exit policy for the selected position
- users can confirm monitoring / notification readiness for that position
- if any per-position activation exists, it must be limited to operational readiness, not custom exit logic
- the UX must never imply users can edit the core directional policy or MVP confirmation logic

### Flow 5: Receive Breach Alert

- native push alert is the primary mobile entry point
- alert copy states:
  - affected position
  - breach direction
  - prompt to review exit preview
- tap opens directly into the relevant position / preview context

### Flow 6: Inspect Execution Preview

- preview opens with the directional state first
- users see the exact sequence:
  - remove liquidity
  - collect fees
  - swap direction
  - expected post-exit posture
- preview shows estimate freshness, slippage/risk caveats, and route context without overwhelming detail

### Flow 7: Sign Execution

- user reviews preview
- user explicitly continues to sign
- wallet handoff occurs with strong context continuity
- return path shows submitted / pending / partial / failed / confirmed state clearly

### Flow 8: View Execution Result And Off-Chain History

- execution result screen shows latest terminal or in-flight state
- history screen shows prior triggers, preview refreshes, signature attempts, submissions, reconciliation updates, and outcomes
- users can filter by position and see directional context for each event

### Flow 9: Review Failures Or Retries

- failure screen distinguishes:
  - stale preview before signing
  - submission failure
  - expiry
  - partial completion
  - reconciliation delay
- retry CTA only appears when a full retry is actually allowed
- partial completion states route to explanation and next allowed action, not blind replay

## 4. Navigation Model

### React Native Primary Navigation

- bottom tab navigation with:
  - Positions
  - Alerts
  - History
- Wallet / Settings accessed from header action or profile sheet
- deep links from push notifications bypass tab browsing and open the relevant preview or position directly

### In-Flow Navigation

- Positions list -> Position Detail -> Execution Preview -> Sign -> Execution Result
- Alerts list -> Alert Detail / Execution Preview
- History list -> History Detail

### Navigation Rules

- never force users through a home dashboard before they can act
- never bury actionable breach state behind more than one tap from app entry
- preserve the return path after wallet handoff

## 5. Screen-Level UX Requirements

### Screen: Wallet Connect / First-Time Setup

- concise framing that this is a user-signed exit assistant
- supported wallet connection only
- notification permission request with reason and consequence
- readiness summary:
  - wallet connected
  - supported positions found
  - notifications enabled / degraded

### Screen: Positions List

- compact list of supported Orca positions
- each item shows pool name, in-range / below-range / above-range state, and latest operational state
- actionable positions rise to the top
- no portfolio-wide analytics, charts, or unrelated asset data

### Screen: Position Detail

- top card shows current range status as the first piece of information
- second layer shows what current out-of-range posture means
- exit policy summary is always visible:
  - downside breach below range -> remove liquidity -> collect fees -> swap SOL to USDC
  - upside breach above range -> remove liquidity -> collect fees -> swap USDC to SOL
- show monitoring readiness and latest alert / execution state

### Screen: Alert Detail / Breach Entry

- breach direction must dominate the hierarchy
- explain what happened in plain language
- show immediate next action: review execution preview
- show best-effort notification disclaimer without distracting from the action

### Screen: Execution Preview

- show exact directional path before any amounts
- sequence block order:
  - trigger direction
  - remove liquidity
  - collect fees
  - swap direction
  - post-exit posture
- downside previews must state SOL -> USDC
- upside previews must state USDC -> SOL
- estimates, slippage, route quality, and freshness must be shown in a compact risk panel
- primary CTA is explicit review-and-sign

### Screen: Signing / Submission Status

- maintain context while external wallet UI is active
- after return, show a single authoritative current state
- do not label a submission as complete before reconciliation

### Screen: Execution Result

- top state block shows:
  - confirmed
  - failed
  - expired
  - partial
  - awaiting reconciliation
- show what completed and what did not
- show next allowed action when relevant

### Screen: History List

- reverse-chronological operational log
- each row includes position, direction, event type, and current status
- support position-level filtering
- emphasize clarity over density

### Screen: History Detail

- event timeline with directional context preserved
- show trigger time, notification time if available, preview refreshes, signing attempts, submission, and final reconciliation
- explicitly label this as off-chain operational history, not on-chain proof

## 6. States And Edge Cases

### Core States

- wallet disconnected
- no supported positions found
- in range
- below range, not yet actionable
- above range, not yet actionable
- actionable downside trigger
- actionable upside trigger
- preview fresh
- preview stale
- awaiting signature
- submitted
- confirmed
- failed
- expired
- partial completion
- reconciliation delayed

### Edge Cases

- notification permission denied
- push delivery delayed, but in-app alert available
- mobile app opened outside notification path
- unsupported wallet context on web
- stale preview before signature
- wallet handoff return interrupted
- one or more execution steps completed before failure
- history available but current live quote unavailable

### UX Handling Rules

- every degraded state must explain what the user can still do
- every failure state must state whether retry is allowed
- partial completion must emphasize legibility over optimism
- no edge case should redirect users into a generic dashboard

## 7. Design Priorities For Mobile

- breach direction is the top-level visual priority
- post-exit posture is the second priority
- the action path from alert to preview to sign must fit into short, high-stress sessions
- copy must reduce interpretation burden, not teach CLMM theory
- risk communication must be compact, layered, and always subordinate to the directional action path
- tap targets, typography, and card hierarchy must support one-handed scanning
- primary CTAs must stay singular and obvious on action screens

## 8. PWA-Specific Considerations

- desktop PWA is designed as a useful secondary surface for review, history, and supported manual execution
- desktop PWA can expose more horizontal space for history and event detail, but must keep the same narrow IA
- desktop PWA must not introduce a broader dashboard or analytics shell
- browser notifications are secondary and should not shape the primary UX model
- mobile web and mobile PWA are best-effort surfaces and should degrade honestly when wallet or notification support is weaker than native
- parity is not the goal; continuity of the same core domain model is the goal

## 9. Explicit UX Constraints For The MVP

- no generic wallet information architecture
- no portfolio overview, APY analytics, earnings dashboards, or trading terminal patterns
- no strategy builder, custom rule engine, or automation builder
- no abstraction that hides the directional invariant
- no UI that implies autonomous execution
- no on-chain receipt, attestation, proof, claim, or verification views
- no design decision should optimize desktop browsing at the expense of mobile breach-response speed
- PWA must remain secondary in emphasis, layout, and interaction assumptions
- the MVP UX must preserve the exact directional policy:
  - downside breach below range -> remove liquidity -> collect fees -> swap SOL to USDC
  - upside breach above range -> remove liquidity -> collect fees -> swap USDC to SOL
- the MVP UX must preserve user-signed execution only and off-chain execution history only
