# Deployment Design Plan: Web Static Release + Mobile Store Release

## Purpose

Define a repeatable, repo-aligned deployment design for CLMM V2 that covers:

- web deployment as a static Expo export
- mobile deployment through Expo/EAS build and store submission

This is a design document only. It does not include an implementation plan.

## Scope

### In Scope
- Web build/export workflow
- Web hosting assumptions
- Mobile build workflow
- Mobile store submission workflow
- Required config additions
- Release validation and failure handling
- Environment/secrets expectations

### Out of Scope
- CI/CD automation details
- Backend deployment design
- App runtime changes
- Store listing copy and marketing assets
- Any implementation task breakdown

## Current Repo State

The repo already supports:
- Expo app configuration in `apps/app/app.json`
- Web export via `apps/app/package.json` using `expo export`
- Static web output via `web.output = "static"`

The repo does not yet define:
- `eas.json`
- iOS bundle identifier
- Android package name
- store submission configuration
- release environment conventions

## Design Summary

### Web
The web app will be deployed as a static site generated by Expo export.

### Mobile
The mobile app will be built and submitted with Expo Application Services (EAS), producing signed iOS and Android binaries for store distribution.

## Workflow 1: Web Deployment

### Build
Use the existing web build command:

```bash
pnpm --filter @clmm/app build
```

This generates the static output under:

```text
apps/app/dist
```

### Host Requirements
The web host must support:
- static file hosting
- client-side routing fallback
- HTTPS
- custom domains
- cache invalidation or versioned asset deployment

### Runtime Configuration
Web deployment must define:
- public BFF/API base URL
- any web-safe public configuration values
- environment-specific release values

### Release Flow
1. Run repo checks.
2. Build the web export.
3. Publish `apps/app/dist` to the static host.
4. Verify routing, assets, and API connectivity on the production URL.

## Workflow 2: Mobile Deployment

### Build System
Use EAS Build for native release artifacts.

### Required Configuration
The project needs:
- `eas.json`
- iOS bundle identifier
- Android package name
- production build profile
- submission profile or submission settings

### Store Submission
Use EAS Submit, or equivalent store submission flow, to push builds to:
- App Store Connect for iOS
- Google Play Console for Android

### Release Flow
1. Configure native identifiers and EAS profiles.
2. Authenticate Expo/EAS.
3. Build iOS and Android release binaries.
4. Validate the resulting artifacts.
5. Submit to the stores.
6. Verify release status and metadata.

## Environment and Secrets

### Web
Likely needs:
- public API base URL
- any browser-safe feature flags or public keys

### Mobile
Likely needs:
- EAS credentials
- Apple developer credentials
- Google Play service account credentials
- deep link / scheme configuration
- build metadata for versioning

### Secret Rules
- No secrets committed to git
- Secrets stored in CI, EAS, or host-managed secret storage
- Public env values remain explicit and documented

## Versioning

### Web
- Web deployments can happen on any release cadence
- Web versioning should still remain aligned with app release metadata

### Mobile
- iOS and Android require monotonic version/build numbers
- Store submission must use platform-correct version metadata

## Validation Gates

Before release, the following should pass:
- install reproducibility
- typecheck
- lint
- dependency boundary scan
- test suite
- web export build
- mobile config sanity checks

For mobile releases, also verify:
- app launches on device/emulator
- wallet/session flows still work in production build
- deep links resolve correctly

## Failure Handling

### Web
If deployment fails:
- do not promote partial assets
- keep the previous release live
- re-export and redeploy after fixing the issue

### Mobile
If build or submission fails:
- do not mark the release as shipped
- keep failure logs and artifact references
- rebuild after fixing config or credential issues

## Recommended Deployment Model

Use:
- static hosting for the web export
- EAS Build + EAS Submit for mobile
- explicit native identifiers in app config
- separate but documented web and mobile release flows

## Open Decisions

Before implementation, decide:
- which static host will serve the web app
- whether mobile should go through TestFlight / Internal Testing first
- whether staging and production should be separate environments
- where release-time environment variables live
- whether releases will be manual or scripted later

## Acceptance Criteria

This design is complete when:
- the web deployment path is defined from build to hosting
- the mobile deployment path is defined from build to store submission
- required config additions are identified
- release validation and failure handling are documented
- the workflow is specific enough to hand off for implementation planning
