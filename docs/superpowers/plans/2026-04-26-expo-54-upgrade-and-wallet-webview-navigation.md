# Expo SDK 54 Upgrade and Wallet WebView Navigation — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Upgrade `apps/app` to Expo SDK 54.x while staying on Legacy Architecture, expose the wallet WebView navigation strategy as a typed constant, run the validation matrix, and record the resulting Outcome A/B/C decision in an ADR — all in one PR.

**Architecture:** Run `npx expo install --fix` to derive the dependency set rather than hand-pinning versions. Add `WALLET_WEBVIEW_NAVIGATION_STRATEGY` to `webNavigation.ts` so `navigateRoute()` is strategy-driven instead of hard-coded UA-detection. Validate against Phantom mobile + Desktop Chrome+extension as merge gates, capture evidence in `docs/decisions/0001-wallet-webview-navigation.md`, and apply the chosen outcome inside the same PR.

**Tech Stack:** Expo SDK 54.x, Expo Router 5.x, React Native 0.81.x (resolver-driven), Reanimated 4.x or compatible (resolver-driven), pnpm 9, TypeScript 5.x, Vitest 1.6.x, NativeWind 4.x.

**Spec:** `docs/superpowers/specs/2026-04-26-expo-54-upgrade-and-wallet-webview-navigation-design.md` (commit `0293a8f`).

**Issue:** [opsclawd/clmm-v2#39](https://github.com/opsclawd/clmm-v2/issues/39).

## Abort Triggers (apply throughout)

If any of the following is hit and cannot be resolved within the SDK bump scope, stop the plan and run the **Rollback Procedure** (see end of plan):

1. Reanimated incompatibility — no SDK 54 compatible Reanimated version supports Legacy Architecture.
2. `expo-doctor` reports a critical issue with no resolution path inside the SDK bump scope.
3. `@solana/connector`, `@solana-mobile/*`, or wallet-standard packages require breaking source changes to consume.
4. NativeWind / Tailwind compatibility break that requires rewriting style code beyond config tweaks.
5. `pnpm test` cannot be made green without modifying application code unrelated to the SDK bump.
6. New Architecture is hard-required by the SDK 54 toolchain.
7. Static web build / Cloudflare deployment cannot be restored without changing deployment architecture.

- [ ] **Step 3: Refactor `navigateRoute()` to dispatch on the constant**

Replace the current `navigateRoute` function (lines 67–85 of `webNavigation.ts`) with:

```ts
export function navigateRoute(params: {
  router: RouterLike;
  path: string;
  method: NavigationMethod;
}): void {
  const canonicalPath = normalizeExpoRouterRoute(params.path);

  switch (WALLET_WEBVIEW_NAVIGATION_STRATEGY) {
    case 'soft-preferred':
      navigateSoftPreferred(params.router, canonicalPath, params.method);
      return;
    case 'hard-fallback':
      navigateHardFallback(params.router, canonicalPath, params.method);
      return;
    case 'capability-driven':
      navigateCapabilityDriven(params.router, canonicalPath, params.method);
      return;
  }
}

function navigateSoftPreferred(router: RouterLike, canonicalPath: string, method: NavigationMethod): void {
  if (isWebPlatform() && isSolanaMobileWebView()) {
    hardNavigate(canonicalPath, method);
    return;
  }
  if (method === 'replace') {
    router.replace(canonicalPath);
    return;
  }
  router.push(canonicalPath);
}

function navigateHardFallback(router: RouterLike, canonicalPath: string, method: NavigationMethod): void {
  if (isWebPlatform() && isSolanaMobileWebView()) {
    hardNavigate(canonicalPath, method);
    return;
  }
  if (method === 'replace') {
    router.replace(canonicalPath);
    return;
  }
  router.push(canonicalPath);
}

function navigateCapabilityDriven(router: RouterLike, canonicalPath: string, method: NavigationMethod): void {
  if (typeof console !== 'undefined' && typeof console.warn === 'function') {
    console.warn(
      "[webNavigation] strategy 'capability-driven' is a stub; falling back to 'hard-fallback' behavior. " +
        'See docs/decisions/0001-wallet-webview-navigation.md.',
    );
  }
  navigateHardFallback(router, canonicalPath, method);
}
```

`'soft-preferred'` and `'hard-fallback'` have identical behavior at this point because the current behavior already prefers soft navigation for non-WebView contexts. Outcome A's *real* difference — preferring soft-nav inside the WebView — is applied in Task 11 only if the matrix evidence supports it. This task only locks in the dispatch shape.
